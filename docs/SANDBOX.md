# Scorch (CourseFuzz) Security & Sandbox Architecture

**Last Updated:** July 22, 2026  
**Audience:** Security Auditors, Technical Hackathon Judges, Infrastructure Engineers  

---

## Security Model Overview

Scorch (CourseFuzz) executes Python code to verify hypotheses generated by LLMs against reference solutions and misconception programs. Executing untrusted or dynamically generated code requires strict containment to prevent arbitrary code execution (RCE), host filesystem tampering, or infinite-loop resource exhaustion attacks.

Scorch operates on a **tiered security boundary** architecture:

```
[ Tier 1: Local Restricted Runner ]  ===>  [ Tier 2: Isolated Process Runner ]  ===>  [ Tier 3: gVisor / Docker Container Sandbox ]
  (Development / Local Demo)                 (Subprocess setrlimit RAM/Time)           (Production Multi-Tenant Cloud)
```

---

## 1. Local Restricted Runner (Development / Demo Mode)

In local demo mode (`LocalRestrictedRunner`), code is constrained at both the AST syntax level and runtime execution level:

### A. AST Syntax Whitelist (`runner.py`)
Before execution, every program is parsed into an Abstract Syntax Tree (AST). The AST visitor enforces a strict node whitelist (`ALLOWED_NODES`).
- **Permitted Syntax**: Basic arithmetic (`+`, `-`, `*`, `%`, `**`), comparison operators (`==`, `!=`, `<`, `>`, `<=`, `>=`), conditional branches (`if`, `if/else`), loop control (`for`, `while`), subscript access (`memo[n]`), and safe recursion.
- **Prohibited Syntax**: Import statements (`import os`, `import sys`), file I/O, `eval`, `exec`, process creation, and network calls.

### B. Builtins Isolation
Execution occurs within an isolated global scope (`globals_dict`) where `__builtins__` is restricted strictly to safe primitives:
`range`, `len`, `min`, `max`, `abs`, `sum`, `int`, `str`, `bool`, `list`, `dict`, `set`, `tuple`.

---

## 2. Subprocess Resource Limits (`sandbox.py`)

When running in subprocess isolation mode (`sandbox.py`), OS-level process limits are enforced via `resource.setrlimit`:
- **Memory Ceiling**: Hard-capped to **128 MB RAM** per process (`RLIMIT_AS`). Memory-exhaustion loops (`'a' * 10**9`) fail immediately with `MemoryError`.
- **CPU Wall Deadline**: Hard-capped to **1.0â€“2.0 seconds** CPU wall-clock time. Infinite loops (`while True: pass`) are forcefully terminated with `subprocess.TimeoutExpired`.
- **Output Ceiling**: Captured stdout/stderr is truncated to **4 KB max** to prevent buffer flooding.

---

## 3. Production Container Isolation (`GVisorDockerRunner`)

For production multi-tenant cloud deployments, Scorch integrates with **gVisor (Google's sandboxed container runtime)** via `GVisorDockerRunner`:
- **Kernel Isolation**: User code runs inside a gVisor `runsc` sandbox, intercepting Linux syscalls in user space.
- **Network Disabling**: `--network=none` disables all inbound and outbound TCP/UDP socket creation.
- **Read-Only Root Filesystem**: `--read-only` prevents any disk modification.
- **Seccomp Filters**: Restricts system calls to a minimal execution subset.

---

## Summary

| Defense Layer | Mechanism | Threat Protected Against |
| :--- | :--- | :--- |
| **AST Parser** | `ALLOWED_NODES` Whitelist | Malicious `import os`, `eval()`, disk/network calls. |
| **Builtins Sandbox** | `__builtins__` Dictionary Restricting | Access to system internals or process inspection. |
| **OS Resource Limits** | `resource.setrlimit` + Timeouts | Memory-exhaustion attacks & infinite loops. |
| **Container Runtime** | `gVisor (runsc)` + `--network=none` | Host kernel exploits & lateral network pivot attacks. |
